# HireLine &mdash; Job Board & Applicant Tracking System (ATS)

A full-stack Django capstone project: a job board where candidates can browse,
search, and apply to jobs without an account, and a recruiter-only Applicant
Tracking System (ATS) with a dashboard and Kanban-style hiring pipeline.

## Tech stack

- **Backend:** Python, Django, Django ORM, ModelForms, Class-Based Views
- **Frontend:** HTML5, CSS3 (custom design system), JavaScript, Bootstrap 5, Bootstrap Icons
- **Database:** SQLite (development), PostgreSQL (production)
- **Deployment:** Render, Gunicorn, WhiteNoise

## 1. Local setup (VS Code)

```bash
# 1. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Apply database migrations (creates db.sqlite3 locally)
python manage.py migrate

# 4. Create a recruiter/admin account
python manage.py createsuperuser

# 5. (Optional) seed a few demo companies & jobs so the board isn't empty
python manage.py seed_demo_data

# 6. Run the development server
python manage.py runserver
```

Visit:
- **Public job board:** http://127.0.0.1:8000/
- **Recruiter login:** http://127.0.0.1:8000/recruiter/login/
- **Django Admin:** http://127.0.0.1:8000/admin/

No environment variables are required for local development &mdash; the
project defaults to `DEBUG=True` and a local SQLite database.

### Alternative: automatic superuser creation

Instead of `createsuperuser`, you can set environment variables and run
the provided script (this is the same mechanism used in production):

```bash
export ADMIN_USERNAME=admin
export ADMIN_EMAIL=admin@example.com
export ADMIN_PASSWORD="a-strong-password"
python create_superuser.py
```

This only creates the user if it doesn't already exist, and never prints
or overwrites a password.

## 2. Implemented features

**Job seekers (public, no login required)**
- Browse active job postings with responsive Bootstrap 5 cards
- Server-side search/filter by keyword, skill, location, and job type (GET params)
- Real-time client-side filtering of cards already on the page (no reload)
- Job detail page with full company/job information
- Apply via a modal form (client + server-side validation)
- Duplicate applications (same email + same job) are blocked at both the
  application layer and the database (`UniqueConstraint`)

**Recruiters (staff-only, Django auth)**
- Login / logout
- Dashboard with summary stats (total, received, under review, shortlisted, rejected)
- Filter applications by job, candidate name/email, and status
- Kanban-style ATS pipeline (Applied → Under Review → Shortlisted → Rejected)
- Secure, POST-only, CSRF-protected, staff-only status updates via `fetch()`

**Administrators (Django Admin)**
- Full CRUD for companies and job postings (activate/deactivate jobs)
- Search/filter for candidates and applications, with `list_select_related`
  and `autocomplete_fields` for performance

**Cross-cutting**
- Dark mode toggle (persisted in `localStorage`), applied across navbar,
  cards, forms, modals, dashboard, Kanban board, tables, and footer
- Pagination on job listings and the applications table
- `select_related()` used throughout to avoid N+1 queries

## 3. Running tests / manual verification checklist

After any change, run:

```bash
python manage.py check
python manage.py test        # if you add test cases
python manage.py runserver
```

Manually verify:
1. Public job browsing, search, and filtering (with and without JS enabled)
2. Job application submission and validation messages
3. Duplicate application prevention (same email + same job)
4. Applying to a different job with the same email (should succeed)
5. Recruiter login redirect when logged out; permission denial for
   non-staff users; dashboard access for staff users
6. ATS status transitions: Applied → Review → Shortlist → Rejected
7. Status update rejected on GET requests and without CSRF/staff auth
8. Dark mode persists after navigating and reloading

## 4. Production configuration

Set these environment variables in production (Render sets most of these
automatically via `render.yaml`):

| Variable | Purpose |
|---|---|
| `SECRET_KEY` | Django secret key (generated by Render) |
| `DEBUG` | Must be `False` in production |
| `ALLOWED_HOSTS` | Comma-separated list of allowed hostnames |
| `CSRF_TRUSTED_ORIGINS` | Comma-separated list of trusted origins (with scheme) |
| `DATABASE_URL` | PostgreSQL connection string (Render provides this) |
| `ADMIN_USERNAME` / `ADMIN_EMAIL` / `ADMIN_PASSWORD` | Optional automatic superuser creation |

Static files are served by **WhiteNoise** using
`CompressedManifestStaticFilesStorage`; run
`python manage.py collectstatic` as part of the build step (already wired
into `render.yaml`).

## 5. Deploying to Render

1. Push this project to a GitHub repository (make sure `.env`, `db.sqlite3`,
   and `staticfiles/` are **not** committed &mdash; see `.gitignore`).
2. In Render, choose **New +** → **Blueprint**, and point it at this repo.
   Render will read `render.yaml` and provision:
   - A **PostgreSQL** database (`jobboard-ats-db`)
   - A **web service** that installs dependencies, runs
     `collectstatic` and `migrate` on build, then creates the superuser
     (if `ADMIN_USERNAME`/`ADMIN_PASSWORD` are set) and starts Gunicorn
     with a low worker count suitable for free-tier RAM.
3. In the Render dashboard, set the `ADMIN_USERNAME`, `ADMIN_EMAIL`, and
   `ADMIN_PASSWORD` environment variables (marked `sync: false` so they
   aren't stored in the blueprint file itself).
4. Deploy. Your app will be available at the generated `*.onrender.com` URL.

Because Render's web-service filesystem is ephemeral, the project never
relies on SQLite in production &mdash; `DATABASE_URL` (PostgreSQL) is
required whenever it is set, and only falls back to SQLite when it is
completely absent (i.e., local development).

## 6. Security notes

- CSRF protection is enabled globally and used on every state-changing form/request
- All state-changing endpoints (apply, status update) are POST-only
- Recruiter dashboard and status-update endpoint require `is_staff`
- Duplicate applications are prevented at the form/view layer *and* enforced
  by a database `UniqueConstraint` (with `IntegrityError` handled safely)
- No secrets, passwords, or keys are committed to the repository
